# Design — pattern_library Structural Matching Fix (pattern_key)

## Architecture Decision

One new migration (`supabase/migrations/`) adding a nullable `pattern_key` column + index to
`pattern_library`, plus a small, pure `buildPatternKey()` helper in `learning.ts` that replaces the
description-equality lookup in `updatePatternLibrary()`. STEP 0 verification surfaced one
necessary plumbing detail beyond the originating request's explicit file list: `db.ts`'s
`upsertPattern()` and `getPatternLibrary()` currently don't write or read a `pattern_key`
column at all — without touching them, the new column would exist in the schema but never
actually be persisted or read back, making the whole fix a no-op. This is included as the minimal
necessary wiring, not scope creep — `TradingPattern`'s TS type also needs the new optional field
for the same reason.

## Data Flow

1. `evaluateClosedTrade()` computes `evaluation.stateFingerprint` exactly as today (unchanged,
   `learning.ts:93-104`).
2. `updatePatternLibrary()` computes `const key = buildPatternKey(evaluation.stateFingerprint)` —
   a pure function returning either a deterministic `"{signal_type}|{z_bucket}|{adx_bucket}|{macd_bucket}"`
   string or `null`.
3. The existing-pattern lookup changes from `library.find(p => p.description === patternDescription
   && p.action === action)` to `library.find(p => key !== null && p.patternKey === key && p.action
   === action)` — the `key !== null` guard ensures a null key can never match any row (including
   another row that also happens to have a null `patternKey`), satisfying FR-05/FR-06 in one
   condition.
4. If matched: the existing merge logic (`sampleCount += 1`, `winCount`/`winRate`/`avgPnLPct`
   recompute, `exampleReasoning` refresh) runs completely unchanged — only the condition that
   selects `existing` changes, not what happens once it's found.
5. If not matched: a new row is created exactly as today, with one addition — `patternKey: key` is
   now set on the object (previously this field didn't exist).
6. `upsertPattern()` (`db.ts`) gains one new mapped column (`pattern_key: pattern.patternKey ??
   null`); `getPatternLibrary()` gains the corresponding read-back
   (`patternKey: row.pattern_key ?? null`). Every other column mapping in both functions is
   unchanged.
7. `description`/`conditions` continue to be generated by the same Claude post-mortem call and
   stored/displayed exactly as today — they simply stop being read inside the match lookup.

## Alternatives Considered

| Option | Pros | Cons | Decision |
|--------|------|------|---------|
| Dedicated `pattern_key text` column + index (proposed) | Clean, queryable, indexable; matches this project's existing migration style for exactly this kind of addition (`state_fingerprint` on `trade_evaluations` was added the same way) | Requires a migration | **Chosen** — explicitly decided by Amaury |
| Embed the composite key inside the existing `conditions` JSONB column, no migration | Avoids a schema change | Conflates a matching key with a field whose declared TS shape (`TradingPattern['conditions']`) is unrelated optional number/boolean fields; not indexable without a JSONB expression index; less discoverable/queryable | Rejected — diagnostic identified this as viable but Amaury chose the dedicated-column path |
| Match on `p.signalType === fp.signal_type` plus separately-stored bucket columns (4 discrete columns instead of 1 composite key) | More granular for future querying by individual bucket | Larger schema surface for no matching-behavior benefit — the composite string does the exact same equality check; more columns to keep in sync | Rejected — unnecessary complexity for this prompt's goal |
| Update `db.ts` to wire the new column through | Required — without it, the column is inert | N/A | **Included** — necessary plumbing, not a separate design choice |

## Impact on Existing Files

| File | Change Type | Description |
|------|------------|--------------|
| `supabase/migrations/{timestamp}_add_pattern_key_to_pattern_library.sql` | CREATE | `ALTER TABLE pattern_library ADD COLUMN IF NOT EXISTS pattern_key text;` + `CREATE INDEX IF NOT EXISTS idx_pattern_library_pattern_key ON pattern_library (pattern_key);` |
| `src/lib/learning.ts` | MODIFY | Add exported `buildPatternKey()` helper; change `updatePatternLibrary()`'s match condition to use it instead of `description` equality; set `patternKey` on newly-created rows. |
| `src/lib/types.ts` | MODIFY | Add `patternKey?: string | null` to `TradingPattern`. |
| `src/lib/db.ts` | MODIFY | `upsertPattern()`: add `pattern_key: pattern.patternKey ?? null` to the upsert payload. `getPatternLibrary()`: add `patternKey: row.pattern_key ?? null` to the row-mapping return object. |
| `src/lib/__tests__/*` (new file, name decided at implementation time) | CREATE | New tests: key-based matching (same key merges, different key doesn't), null-fingerprint fallback (always creates new row), the two real XOM trades replayed through `buildPatternKey()` to report their actual key equality, `description` remaining unused for matching. |

## Protected Zone Impact

⚠️ `learning.ts` is modified — treated as Protected Zone per prior session's precedent, explicitly
authorized in the originating request. A new database migration is also added — per `CLAUDE.md`'s
File Permission Matrix ("Any DB migration"), this requires explicit confirmation, likewise already
given. `config.ts`, `claude-agent.ts`, `risk-manager.ts`, `indicators.ts` are not touched.

## Database Changes

One new column: `pattern_library.pattern_key` (`text`, nullable). One new index:
`idx_pattern_library_pattern_key`. No data migration, no row updates, no RLS policy change (adding
a nullable column does not require one).

## Open Questions

None blocking. The `db.ts`/`types.ts` plumbing needed to make the new column functional was
identified during STEP 0 verification and is included above as necessary implementation detail,
not a decision requiring further Amaury input — it directly follows from the explicitly-authorized
goal ("used as the match key going forward").
